# SQLAlchemy and Alembic Patterns

Use this reference for Python services that use SQLAlchemy 2.x in async mode.
The domain layer defines repository protocols; infrastructure implements them.

## Engine and Session Factory

- Create one async engine per database role.
- Use `async_sessionmaker` for sessions.
- Set `pool_pre_ping=True`.
- Tune pool size from environment variables.
- Use request-scoped sessions in FastAPI dependencies.

```python
from sqlalchemy.ext.asyncio import AsyncSession, async_sessionmaker, create_async_engine

engine = create_async_engine(
    settings.database_url,
    pool_size=settings.db_pool_size,
    max_overflow=settings.db_max_overflow,
    pool_pre_ping=True,
)

SessionFactory = async_sessionmaker(
    bind=engine,
    class_=AsyncSession,
    expire_on_commit=False,
)
```

## Declarative Models

- Use SQLAlchemy 2.x `Mapped` and `mapped_column`.
- Keep ORM models in infrastructure.
- Provide explicit `to_domain()` and `from_domain()` mapping methods.
- Use naming conventions so Alembic generates stable constraint names.

```python
class OrderModel(Base):
    __tablename__ = "orders"

    id: Mapped[UUID] = mapped_column(PgUuid(as_uuid=True), primary_key=True)
    customer_id: Mapped[UUID] = mapped_column(PgUuid(as_uuid=True), nullable=False)
    status: Mapped[str] = mapped_column(String(40), nullable=False)
    total: Mapped[Decimal] = mapped_column(Numeric(12, 2), nullable=False)
    version: Mapped[int] = mapped_column(Integer, nullable=False, default=0)
```

## Repository Pattern

- Domain defines protocols.
- Infrastructure accepts `AsyncSession`.
- Application handlers decide transaction boundaries through a unit of work.
- Avoid leaking ORM models to the application layer.

## Unit of Work

- Open one session per request or message.
- Commit once after all aggregate changes and outbox rows are staged.
- Roll back on exception.
- Do not publish Kafka messages inside the transaction.

```python
class SqlAlchemyUnitOfWork:
    def __init__(self, session: AsyncSession):
        self.session = session
        self.orders = SqlAlchemyOrderRepository(session)

    async def __aenter__(self) -> SqlAlchemyUnitOfWork:
        return self

    async def __aexit__(self, exc_type, exc, traceback) -> None:
        if exc is not None:
            await self.session.rollback()

    async def commit(self) -> None:
        await self.session.commit()
```

## Relationships

- Prefer `lazy="selectin"` for async relationship loading.
- Avoid lazy loads that perform hidden I/O during mapping.
- Keep aggregate boundaries explicit.
- Use association tables for many-to-many relationships.

## Query Patterns

- Use `select()` instead of legacy query APIs.
- Use explicit joins for read models.
- Use CTEs or window functions for reporting queries when the contract needs
  them.
- Keep read-model SQL in query handlers or dedicated read repositories.

## Soft Delete

- Add `is_deleted`, `deleted_at`, and `deleted_by` where deletion is logical.
- Apply filters in repository methods or with explicit statement helpers.
- Do not hide soft-delete behavior where audit or restore workflows need
  precise control.

## Audit Fields

- Use database defaults for `created_at` and `updated_at` when possible.
- Use request context for `created_by` and `updated_by`.
- Keep audit fields consistent across aggregates.
- Treat audit history and timeline events as append-only.

## Optimistic Concurrency

- Add integer `version` columns to aggregates that need conflict detection.
- Check version in update statements.
- Raise a domain/application conflict error on mismatch.
- Map conflicts to HTTP 409 problem responses.

## Alembic Setup

- Use async `env.py`.
- Import all model metadata in Alembic configuration.
- Enable naming conventions.
- Review autogenerated migrations before committing.
- Keep migration IDs descriptive.

```python
async def run_migrations_online() -> None:
    connectable = create_async_engine(config.get_main_option("sqlalchemy.url"))
    async with connectable.connect() as connection:
        await connection.run_sync(do_run_migrations)
    await connectable.dispose()
```

## Zero-Downtime Migrations

Use expand-migrate-contract:

1. Add nullable columns or new tables.
2. Deploy code that writes both old and new shapes.
3. Backfill data in batches.
4. Deploy code that reads the new shape.
5. Remove old columns after the fleet is safe.

Never combine destructive schema changes with code that still depends on the
old shape.

## Seed Data

- Use Alembic data migrations only for required reference data.
- Use fixture scripts for demo or local data.
- Make seed scripts idempotent.
- Do not put secrets in seed data.

## Connection Pooling

- Set `pool_size`, `max_overflow`, `pool_timeout`, and `pool_recycle`.
- Use `pool_pre_ping=True` for long-lived services.
- Keep worker concurrency aligned with database capacity.
- Monitor checkout time and connection errors.

## Read Replicas

- Route reads to replicas only when stale reads are acceptable.
- Keep writes and read-after-write queries on the primary session.
- Do not use replicas for authorization decisions that require fresh state.

## Transactional Outbox

- Store events in `outbox_messages` in the same transaction as aggregate
  changes.
- Relay workers poll unpublished rows with `FOR UPDATE SKIP LOCKED`.
- Mark rows as published after message broker acknowledgement.
- Deduplicate consumers by `event_id`.
- Clean up old published rows after retention.

## JSONB

- Store flexible metadata in JSONB only when the contract needs it.
- Validate JSONB payloads with Pydantic before persistence.
- Add GIN indexes for queryable paths.
- Avoid hiding core domain fields inside JSONB.

## Full-Text Search

- Use generated `tsvector` columns for high-volume search.
- Use GIN indexes.
- Keep search ranking and filters in read models.
- Test query plans for large datasets.

## Enums

- Prefer Python `Enum` in domain/application code.
- Use PostgreSQL enum types only when operationally justified.
- Plan enum value additions as migrations.
- Do not rename enum values without a compatibility plan.

## Custom Types

- Use `TypeDecorator` for money, encrypted values, and normalized identifiers.
- Keep conversion deterministic.
- Write tests for round-trip serialization.
- Avoid custom types when simple columns are clearer.

## Event Listeners

- Use SQLAlchemy event listeners sparingly.
- Prefer explicit application behavior for business rules.
- Event listeners are acceptable for audit timestamps and technical defaults.
- Do not publish domain events from SQLAlchemy lifecycle hooks.

## Testing

- Use PostgreSQL testcontainers for integration tests.
- Use rollback-per-test fixtures for repository tests.
- Run Alembic migrations in integration test startup.
- Test repository mapping in both directions.

## Bulk Operations

- Use bulk insert/update for administrative or import workflows only.
- Preserve domain invariants and audit behavior.
- Chunk large writes.
- Track failed rows with stable identifiers.

## Multi-Tenancy

- Use discriminator columns unless architecture explicitly chooses
  schema-per-tenant.
- Include tenant filters in repository methods.
- Enforce tenant boundaries in authorization and persistence.
- Add composite indexes for tenant plus common lookup fields.
